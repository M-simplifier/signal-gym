## Randomization Notes

- Do not cycle through cases in array order. Derive a local session seed from `(todayKey, mode, drillLevel, sessionCount)` plus a small per-round counter, then choose cases with a deterministic shuffle. This preserves privacy because it uses only existing local state.
- Combine `domain` and load by keeping a short local recent-case buffer in `localStorage`: last case ids, last domains, and last correct answers. When selecting a round, down-rank cases from the same domain, same correctIndex, or same prompt pattern used in the last few rounds.
- Use load to choose item density, not factual ambiguity. Low load can prefer fewer numbers and simpler timelines; high load can prefer tighter distractors, more entities, or multiple status flags. Correct answers must remain explicit in the source or passage.
- Add a tiny deterministic tie-breaker from the seed so two users with the same level do not see the same order, while repeat sessions on the same day still feel stable enough to debug.
- Keep all randomization client-side. No telemetry, backend, account id, remote model call, or external content fetch is needed.

## Quality Checklist

- Each case is synthetic and objective.
- No real current companies, named people, private data, or advice requiring external verification.
- Correct answer is unambiguous from the provided source or passage.
- Source and passage lengths stay roughly within the existing app range.
- Distractors are plausible but clearly wrong for a stated reason.
- Domains are mixed across rounds; no single office-software template dominates.
- Claim Gate focuses on contradiction, overstatement, unsupported inference, or policy-boundary risk.
- Dense Read asks for retained facts, conditions, ordering, or definitions after the passage is hidden.
- Rationale explains the evidence boundary without adding new facts.
- ASCII-only text; safe for current English UI and local-first distribution.
